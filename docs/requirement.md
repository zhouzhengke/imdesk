# IM工单系统需求文档

## 1. 项目背景

当前客服与技术支持团队需要在多个即时通讯（IM）渠道（企业微信、飞书、钉钉、Slack 等）上处理来自不同合作方（资方）的用户咨询与问题反馈。现有方式依赖人工值守、手动记录，存在响应不及时、问题跟踪困难、跨渠道信息割裂等问题。

因此需要建设一套 IM工单系统，实现：

- 多渠道统一接入，自动识别资方身份
- 智能值班机制，保证消息必有人响应
- 用户问题自动转为工单，支持简单查询与流转
- 引入大模型能力，实现智能路由、知识库问答、工单辅助处理

## 2. 产品目标

- **统一收口**：企业微信、飞书等渠道的资方对话统一进入工单系统
- **智能路由**：规则引擎（毫秒级）优先 + 大模型兜底，自动识别用户意图并分流
- **自动化工单**：用户发送的问题自动创建工单，并关联渠道、资方、会话上下文
- **知识库智能问答**：FAQ 精确匹配 → 文档 RAG 检索 → 历史工单检索，三层知识源自动回复，减少人工工单量
- **智能值班**：根据预设规则自动分配待处理消息给值班人员，避免消息遗漏
- **便捷查询**：允许用户（资方或内部客服）通过自然语言或指令查询工单状态、常见信息

## 3. 用户角色

| 角色 | 描述 | 主要操作 |
|------|------|---------|
| 资方用户 | 合作金融机构/资金方的工作人员 | 发起咨询、查询工单进度、获取知识库自动回复 |
| 客服/技术支持 | 处理工单的内部人员 | 接收、回复、转交、驳回、延期、关闭工单 |
| 系统管理员 | 管理后台配置 | 配置渠道、值班规则、工单模板、知识库、通知模板 |
| 值班负责人 | 排班管理与消息调度 | 设置值班表、监控未分配消息、手动指派 |

## 4. 功能需求

### 4.1 多渠道对接

| 渠道 | 支持能力 | 备注 |
|------|---------|------|
| 企业微信 | 接收单聊/群聊消息；发送文本、图文、文件；获取用户身份（CorpId+UserId） | 优先支持 |
| 飞书 | 接收 im 消息；发送消息卡片；获取 open_id/tenant_key | 首批支持 |
| 钉钉 | 接收机器人回调；发送 Markdown/文本 | 二期支持 |
| Slack | 接收斜杠命令或事件订阅；发送消息 | 二期支持 |

功能点：
- 各渠道通过官方开放 API 或 Webhook 接入，统一转换为系统内部消息格式
- 能够根据渠道来源、会话对象（用户ID）自动识别所属"资方"
- 企业微信：根据外部联系人标签、CorpId+UserId 或群聊 ID 识别
- 飞书：根据租户 tenant_key、部门或自定义字段识别
- 支持一个资方对应多个渠道入口，工单数据全局统一

### 4.2 智能路由

采用「规则优先 + 大模型兜底」策略：

**规则引擎（毫秒级）**：
- 前缀匹配：`#查单`、`#我的工单`、`#帮助` 等
- 关键词匹配：`转人工`、`联系客服`、`创建工单` 等
- 正则匹配：工单编号格式 `IM-\d{8}-\d{4}`

**大模型兜底**：
- 规则未命中时调用 LLM 做意图识别
- 返回结构化输出：`{intent, confidence, entities, priority, sentiment}`
- 意图类型：工单查询 / 创建工单 / 补充说明 / 知识库问答 / 业务查询 / 寒暄 / 紧急升级

**升级策略**（KB 问答 → 自动创建工单）：
1. 用户主动要求转人工（最高优先级）
2. LLM 判断 KB 无法回答（置信度低于阈值或用户情绪负面）
3. KB 交互轮次超限（默认 ≥ 3 轮）
4. LLM 直接识别为创建工单意图

### 4.3 知识库与智能问答

三层检索架构：

| 层级 | 数据源 | 检索方式 | 响应速度 |
|------|--------|---------|---------|
| L1 | FAQ 库（MySQL） | 精确/模糊匹配 | 毫秒级 |
| L2 | 文档库（ES 向量索引） | 向量检索 + LLM 生成 | ~500ms |
| L3 | 历史工单库（ES 向量索引） | 向量检索 + LLM 生成 | ~500ms |

- 管理员可上传 PDF/Word/Markdown 文档，系统自动切片、向量化
- 工单关闭后 LLM 自动生成摘要和关键词，经人工审核后入库
- 客服工作台展示「相似历史工单 Top3」，供客服参考
- 高频相似工单可一键转为 FAQ

### 4.4 自动值班

目标：保证所有从 IM 渠道进入的用户消息，在规定时间内必定被分配给一位值班人员。

**排班管理**：
- 支持按日/周配置值班表（早班、晚班、周末班）
- 每个班次可设置 1 个主值班 + N 个备选值班
- 值班人员从客服组成员中选择

**消息分配策略（按优先级尝试）**：
1. 优先分配给当前会话上一次处理的客服（上下文继承）
2. 若无可用上次客服，则分配给当前时段值班人员
3. 若值班人员离线或未响应（可配置超时，默认 5 分钟），则转给备选值班
4. 若无任何值班在线，则进入"待分配池"并触发告警（群通知管理员）
5. 会话接管：当主值班超时未回复时，备选值班可主动"领取"该会话

**在线/离线状态**：Phase 1 使用系统内手动签到（上班/休息），预留 IM 状态 API 对接接口。

### 4.5 工单自动创建

**触发条件**（满足任一即创建工单）：
1. 资方用户发送一条非指令性消息，智能路由识别为创建工单意图，且当前无未关闭的工单关联此会话
2. KB 问答场景触发升级策略
3. 客服手动点击"转为工单"按钮

**工单内容**：

| 字段 | 说明 | 来源 |
|------|------|------|
| 工单ID | 唯一编号，如 IM-20260526-0001 | 系统生成 |
| 渠道 | 企业微信 / 飞书 / 钉钉等 | 自动识别 |
| 资方名称 | 关联的具体资方（如"XX银行"） | 映射配置 |
| 用户ID | 渠道侧用户标识 | 渠道回调 |
| 用户昵称 | 显示名称 | 渠道回调 |
| 问题描述 | 首条消息内容 + 后续补充 | 聊天内容提取 |
| 创建时间 | 首次消息时间 | 系统记录 |
| 当前状态 | 待处理/处理中/已转交/已驳回/已延期/已解决/待确认/已关闭 | 系统维护 |
| 负责客服 | 值班分配的人员 | 系统分配 |
| 优先级 | 普通/紧急/非常紧急 | LLM 识别或客服手动 |
| 标签 | 自动/手动标签 | LLM 自动分类或客服添加 |
| 会话上下文 | 完整对话记录（包括 KB 交互历史） | 消息存储 |

**创建后的行为**：
- 自动回复用户："您的问题已创建工单（编号：XXX），我们将尽快处理。"
- 在客服工作台推送新工单提醒
- 若用户后续继续发言，自动追加到同一工单的"补充说明"中

### 4.6 简单查询能力

支持资方用户在聊天窗口通过自然语言或简单指令查询工单、业务信息。

| 查询示例 | 返回内容 |
|---------|---------|
| `#查单 IM-20260526-0001` | 返回该工单状态、处理人、最近回复 |
| `#我的工单` | 列出当前资方名下最近 5 条工单（编号+状态） |
| `帮助` 或 `?` | 显示可用查询命令列表 |

功能要求：
- 以消息前缀（如 `#`）或自然语言意图识别触发
- 查询结果通过聊天消息返回，格式清晰（表格或列表）
- 对于无法识别的查询，回复："暂不支持该查询，如需帮助请直接描述问题，我们将为您创建工单。"

### 4.7 客服工作台（Web 端）

- **工单列表**：支持按渠道、资方、状态、优先级、负责客服筛选和搜索
- **工单详情**：显示完整对话记录、工单字段，支持内部备注、回复、转交、驳回、延期、关闭、添加标签。侧边栏展示相似历史工单 Top3
- **实时会话**：客服可直接在工作台回复消息，消息双向同步到 IM 渠道
- **值班面板**：查看当前值班表、待分配会话数量、离线告警、手动指派
- **数据统计**：工单量统计、处理时效分析（Phase 1 基础版本）

### 4.8 系统管理

- **资方管理**：维护资方基本信息（名称、对接人、合同期限等）
- **渠道配置**：配置每个渠道下如何识别资方的映射关系
- **值班排班**：配置班次和排班表
- **通知模板**：自定义内外部通知模板
- **知识库管理**：FAQ 维护、文档上传与管理、历史工单审核
- **规则引擎配置**：管理路由规则的关键词和正则

### 4.9 通知系统

**双通道设计**：

| 维度 | 内部通知（→ 客服） | 外部通知（→ 资方用户） |
|------|-------------------|----------------------|
| 通知渠道 | 工作台推送 + IM Bot 私聊 + 群通知 | 回源到原始 IM 渠道 |
| 典型场景 | 新工单分配、超时告警、待分配提醒 | 工单创建确认、状态变更、解决通知 |
| 模板 | 变量占位符 + Markdown | 变量占位符 + 渠道格式适配 |
| 可回复 | 否（单向通知） | 是（用户可继续对话） |

**可变模板参数**：`{{ticket_id}}`、`{{ticket_status}}`、`{{capital_name}}`、`{{user_name}}`、`{{agent_name}}`、`{{category}}`、`{{summary}}`、`{{priority}}`、`{{created_at}}`、`{{resolved_at}}`、`{{channel_name}}`、`{{reply_content}}`

## 5. 非功能需求

| 项目 | 要求 |
|------|------|
| 响应时效 | 用户消息到系统处理完成 ≤ 2 秒；规则引擎路由 ≤ 50ms |
| 消息可靠性 | 消息不丢失，支持重试机制；消息数据保留 ≥ 180 天 |
| 并发能力 | 支持同时处理 1000+ 活跃会话（按单渠道评估） |
| 安全性 | 所有渠道回调需验签；敏感信息需脱敏显示；防范 SQL 注入、XSS |
| 可扩展性 | 预留对接新渠道的接口抽象层；预留 OAuth 用户体系对接接口 |
| 值班高可用 | 分配服务需支持多实例部署，避免单点故障 |

## 6. 约束与假设

- 各 IM 平台均需申请相应的机器人/应用，并拥有发送消息权限
- Phase 1 使用系统内账号体系，预留 OAuth 2.0 对接接口
- 值班人员的在线状态 Phase 1 依赖系统内手动签到
- 钉钉、Slack 渠道为二期支持范围
- 业务系统对接为二期支持范围

## 7. Phase 1 规模

- 10-20 个资方
- 20-50 个客服
- 日工单量 100-500
- 首发渠道：企业微信、飞书

## 附录：交互示例

### 场景一：知识库问答

```
用户在企微群："额度审批流程是什么？"
→ 规则引擎未命中 → LLM 识别 knowledge_query
→ FAQ 命中 → 群内回复："审批流程如下：1. 提交申请 2. 初审..."
```

### 场景二：知识库升级为工单

```
用户追问："我的情况比较特殊，能帮我看下吗？"
→ LLM 识别 intent=create_ticket + sentiment=negative + KB 交互已达 3 轮
→ 自动创建工单 IM-20260526-0001
→ 自动回复："您的问题已创建工单（IM-20260526-0001），请等待回复。"
→ 通知值班客服张三（工作台推送 + IM Bot 私聊）
→ KB 对话历史自动带入工单上下文
```

### 场景三：客服处理工单

```
客服张三在工作台看到新工单 → 点击领取
→ 查看对话记录（包含 KB 交互历史）和相似历史工单 Top3
→ 在群内回复："您好，我来帮您处理..."
→ 消息双向同步：工作台 ↔ 企微群
→ 问题解决后，标记「已解决」→ 通知用户确认
→ 用户确认 / 超时自动关闭
→ 工单关闭 → LLM 生成摘要 → 入库至历史工单库
```
